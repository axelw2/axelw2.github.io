1. Introduction et objectifs du projet

Notre groupe avait pour mission de développer une application web F1 Fantasy League (simulation de gestion d'écuries de Formule 1). L'équipe était composée de :

  • 3 développeurs : responsables du code PHP, des templates HTML/CSS/JS et du schéma SQL.
  • Moi-même : responsable de l'infrastructure Docker, de l'orchestration des conteneurs et de la configuration du réseau interne.

Objectifs de ma partie :

  1. Conteneuriser l'ensemble de la stack : Apache + PHP, MySQL, phpMyAdmin.
  2. Mettre en place un serveur DNS interne (Bind9) pour que les conteneurs communiquent entre eux par noms de domaine (et non par IP fixes).
  3. Permettre aux développeurs de travailler localement avec un environnement identique à la production.
  4. Initialiser automatiquement la base de données avec le schéma fourni par les développeurs.
  5. Assurer la persistance des données (volume pour MySQL).
  6. Documenter l'infrastructure pour que les développeurs puissent l'utiliser sans connaître Docker.

3. Configuration détaillée des services

Dans cette section, je détaille la configuration de chaque service. L'ordre choisi n'est pas anodin : le DNS est configuré en premier car tous les autres services en dépendent pour résoudre les noms de domaine internes.

3.1. Service DNS (Bind9)

Pourquoi un DNS interne ?

Dans une infrastructure multi-conteneurs, on pourrait utiliser des adresses IP fixes. Mais cette approche pose problème : si l'IP d'un conteneur change (par exemple après un redémarrage ou une mise à jour), il faudrait modifier la configuration de tous les autres services. En utilisant un DNS interne, chaque service communique par un nom de domaine (ex: db.f1fantasy-league.com). Les développeurs peuvent donc coder avec ces noms, sans se soucier des adresses IP.

Structure des fichiers de configuration

bind/
├── named.conf
└── zones/
    └── f1fantasy-league.com.db

J'ai créé deux fichiers :

  • named.conf : la configuration générale du serveur Bind9.
  • f1fantasy-league.com.db : le fichier de zone qui contient les enregistrements DNS (correspondances noms → IP).

Explication détaillée de named.conf

options { directory "/etc/bind/zones"; allow-query { any; }; # Autoriser tous les conteneurs à interroger le DNS recursion no; # Désactiver la résolution récursive (pas besoin d'Internet) }; zone "f1fantasy-league.com" { type master; file "/etc/bind/zones/f1fantasy-league.com.db"; };
  • directory : indique où se trouvent les fichiers de zone.
  • allow-query { any; }; : sans cette ligne, seuls les requêtes provenant du conteneur lui-même seraient acceptées. Les autres conteneurs (web, db, phpmyadmin) auraient été bloqués.
  • recursion no : le serveur DNS n'a pas besoin d'aller chercher des noms externes (google.com, etc.). L'application n'utilise que des noms internes. Désactiver la récursion améliore la sécurité.
  • zone "f1fantasy-league.com" : déclare que ce serveur est maître (master) pour cette zone, et que les enregistrements sont dans le fichier spécifié.

Explication détaillée du fichier de zone

$ORIGIN f1fantasy-league.com. $TTL 3600 @ IN SOA ns1.f1fantasy-league.com. admin.f1fantasy-league.com. ( 2026050401 ; Serial (YYYYMMDDNN) 3600 ; Refresh 1800 ; Retry 604800 ; Expire 3600 ; Minimum TTL ) IN NS ns1 ns1 IN A 172.18.0.3 db IN A 172.18.0.2 web IN A 172.18.0.4 admin IN A 172.18.0.5
  • SOA (Start Of Authority) : enregistrement obligatoire qui définit les paramètres de la zone. Le serial (2026050401) est incrémenté à chaque modification du fichier.
  • NS : déclare que ns1 est le serveur de noms pour cette zone.
  • IN A : enregistrements de type A (Address) qui lient chaque nom d'hôte à son adresse IPv4.

Intégration dans Docker Compose

dns: image: ubuntu/bind9 container_name: f1-dns hostname: ns1.f1fantasy-league.com ports: - "53:53/udp" - "53:53/tcp" volumes: - ./bind/named.conf:/etc/bind/named.conf - ./bind/zones:/etc/bind/zones networks: app-network: ipv4_address: 172.18.0.3 command: ["-g", "-c", "/etc/bind/named.conf"]
💡 Explication : Le port 53 (DNS) est exposé sur l'hôte pour faciliter les tests. La commande -g lance Bind9 en mode non-démonisé (le processus reste attaché au conteneur, ce qui est utile pour voir les logs). Le volume monte mes fichiers de configuration personnalisés.

3.2. Service Base de données (MySQL 8.0)

Pourquoi un volume pour les données ?

Par défaut, les conteneurs Docker sont éphémères : quand on les supprime, toutes les données qu'ils contiennent sont perdues. Pour une base de données, c'est inacceptable. J'ai donc créé un volume nommé db_data qui persiste sur le disque de l'hôte. Ainsi, même en supprimant le conteneur MySQL, les données restent intactes.

Configuration Docker du service MySQL

db: image: mysql:8.0 container_name: f1-db hostname: db.f1fantasy-league.com environment: MYSQL_ROOT_PASSWORD: rootpassword MYSQL_DATABASE: formula_one_fantasy_league MYSQL_USER: f1_user MYSQL_PASSWORD: f1_password volumes: - db_data:/var/lib/mysql - ./formula_one_fantasy_league.sql:/docker-entrypoint-initdb.d/init.sql dns: - 172.18.0.3 networks: app-network: ipv4_address: 172.18.0.2 healthcheck: test: ["CMD", "mysqladmin", "ping", "-h", "localhost"] timeout: 20s interval: 10s retries: 10
💡 Explications :
  • Variables d'environnement : Elles initialisent automatiquement la base : création de la base formula_one_fantasy_league, d'un utilisateur f1_user et de son mot de passe. C'est plus propre que de coder ces identifiants en dur dans les fichiers PHP.
  • Initialisation automatique : le fichier formula_one_fantasy_league.sql fourni par les développeurs est copié dans /docker-entrypoint-initdb.d/. À la première création du conteneur, MySQL exécute automatiquement ce script (création des tables, insertion des données de test).
  • Healthcheck : cette vérification périodique permet à Docker de savoir si MySQL est vraiment prêt à accepter des connexions. Le conteneur est considéré comme "healthy" uniquement quand la commande mysqladmin ping réussit. Cette information sera utilisée par phpMyAdmin pour ne démarrer qu'au bon moment.

3.3. Service Web (PHP 8.0 + Apache)

Pourquoi une image PHP avec Apache intégré ?

L'image php:8.0-apache est officielle et très pratique : elle contient à la fois PHP et le serveur web Apache pré-configuré. Pas besoin de créer un conteneur Apache et un conteneur PHP-FPM séparés, ce qui simplifie l'orchestration.

Problème et solution : les extensions MySQL

L'image de base ne contient pas les extensions nécessaires pour se connecter à MySQL (mysqli, PDO, PDO_mysql). J'ai donc ajouté une commande personnalisée qui les installe au démarrage du conteneur.

web: image: php:8.0-apache container_name: f1-web hostname: web.f1fantasy-league.com environment: DB_HOST: db.f1fantasy-league.com DB_NAME: formula_one_fantasy_league DB_USER: f1_user DB_PASSWORD: f1_password volumes: - ./html:/var/www/html ports: - "8080:80" dns: - 172.18.0.3 networks: app-network: ipv4_address: 172.18.0.4 command: bash -c "docker-php-ext-install mysqli pdo pdo_mysql && apache2-foreground"
💡 Explications :
  • Variables d'environnement : Les développeurs peuvent y accéder en PHP via getenv('DB_HOST'). Cela permet d'avoir un code qui fonctionne à la fois en développement (avec Docker) et en production (avec des valeurs différentes).
  • Volume ./html:/var/www/html : Le code PHP est stocké sur l'hôte dans le dossier html/. Les modifications sont prises en compte immédiatement, sans avoir à reconstruire le conteneur à chaque changement. C'est essentiel pour les développeurs.
  • Port 8080:80 : Le port 80 (HTTP) du conteneur est mappé sur le port 8080 de l'hôte. Évite les conflits si un autre service utilise déjà le port 80.

3.4. Service phpMyAdmin

Pourquoi phpMyAdmin ?

Les développeurs de l'équipe n'ont pas tous la même aisance avec SQL. phpMyAdmin offre une interface web pour visualiser et modifier la base de données, ce qui accélère le débogage et les tests.

Dépendance avec MySQL

Un problème classique : phpMyAdmin peut se lancer avant MySQL, ce qui provoque une erreur de connexion. Pour l'éviter, j'utilise deux mécanismes :

  • healthcheck dans le service MySQL (défini plus haut)
  • depends_on avec condition dans phpMyAdmin
phpmyadmin: image: phpmyadmin/phpmyadmin container_name: f1-admin hostname: admin.f1fantasy-league.com environment: PMA_HOST: db.f1fantasy-league.com PMA_USER: root PMA_PASSWORD: rootpassword ports: - "8081:80" dns: - 172.18.0.3 networks: app-network: ipv4_address: 172.18.0.5 depends_on: db: condition: service_healthy
💡 Explications :
  • PMA_HOST : indique à phpMyAdmin quel serveur MySQL utiliser. On utilise le nom DNS db.f1fantasy-league.com grâce à notre serveur Bind9.
  • depends_on ... condition: service_healthy : Docker Compose attendra que MySQL soit passé en état "healthy" (après avoir répondu avec succès au healthcheck) avant de démarrer phpMyAdmin.

4. Orchestration avec Docker Compose

Pourquoi Docker Compose ?

On pourrait lancer chaque conteneur avec une longue commande docker run. Mais cela deviendrait vite ingérable : 4 services, chacun avec ses volumes, ses ports, ses variables d'environnement, et surtout l'ordre de démarrage à respecter.

Docker Compose permet de décrire toute l'infrastructure dans un seul fichier YAML. Une seule commande (docker-compose up -d) lance tous les services en respectant les dépendances.

Comment j'ai construit le fichier

J'ai assemblé les configurations de chaque service (détaillées ci-dessus) dans un seul fichier docker-compose.yml. Voici les éléments importants à retenir :

ÉlémentRôle
version: '3.8'Version du format Compose (supporte les healthchecks et les conditions de dépendance)
services:Déclaration des 4 services (dns, db, web, phpmyadmin)
volumes:Déclaration du volume persistant db_data pour MySQL
networks: app-networkCréation d'un réseau dédié, avec sous-réseau et passerelle définis
dns: 172.18.0.3Présent dans chaque service → force l'utilisation de notre serveur DNS interne
healthcheck + depends_on: condition: service_healthyGarantit l'ordre de démarrage (MySQL avant phpMyAdmin)

Fichier docker-compose.yml complet

version: '3.8' services: web: image: php:8.0-apache container_name: f1-web hostname: web.f1fantasy-league.com environment: DB_HOST: db.f1fantasy-league.com DB_NAME: formula_one_fantasy_league DB_USER: f1_user DB_PASSWORD: f1_password volumes: - ./html:/var/www/html ports: - "8080:80" dns: - 172.18.0.3 networks: app-network: ipv4_address: 172.18.0.4 command: bash -c "docker-php-ext-install mysqli pdo pdo_mysql && apache2-foreground" db: image: mysql:8.0 container_name: f1-db hostname: db.f1fantasy-league.com environment: MYSQL_ROOT_PASSWORD: rootpassword MYSQL_DATABASE: formula_one_fantasy_league MYSQL_USER: f1_user MYSQL_PASSWORD: f1_password volumes: - db_data:/var/lib/mysql - ./formula_one_fantasy_league.sql:/docker-entrypoint-initdb.d/init.sql dns: - 172.18.0.3 networks: app-network: ipv4_address: 172.18.0.2 healthcheck: test: ["CMD", "mysqladmin", "ping", "-h", "localhost"] timeout: 20s interval: 10s retries: 10 phpmyadmin: image: phpmyadmin/phpmyadmin container_name: f1-admin hostname: admin.f1fantasy-league.com environment: PMA_HOST: db.f1fantasy-league.com PMA_USER: root PMA_PASSWORD: rootpassword ports: - "8081:80" dns: - 172.18.0.3 networks: app-network: ipv4_address: 172.18.0.5 depends_on: db: condition: service_healthy dns: image: ubuntu/bind9 container_name: f1-dns hostname: ns1.f1fantasy-league.com ports: - "53:53/udp" - "53:53/tcp" volumes: - ./bind/named.conf:/etc/bind/named.conf - ./bind/zones:/etc/bind/zones networks: app-network: ipv4_address: 172.18.0.3 command: ["-g", "-c", "/etc/bind/named.conf"] volumes: db_data: networks: app-network: driver: bridge ipam: config: - subnet: 172.18.0.0/24 gateway: 172.18.0.1

6. Conclusion – Bilan du travail en groupe

Ce que j'ai apporté à l'équipe

  1. Un environnement reproductible : Chaque développeur, quelle que soit sa machine (Windows, Linux, macOS), obtient exactement la même stack (PHP 8.0, MySQL 8.0, Apache) en une seule commande docker-compose up -d.
  2. Une résolution DNS interne propre : Les développeurs peuvent coder avec db.f1fantasy-league.com au lieu d'IP en dur. Si les IP changent (par exemple après un redémarrage), rien ne casse.
  3. Une séparation claire des couches : Le code PHP est isolé dans html/, la base dans un volume persistant, la configuration DNS dans bind/. Chaque préoccupation a son emplacement dédié.
  4. Un outil d'administration (phpMyAdmin) : Les développeurs non spécialistes en SQL peuvent visualiser et modifier la base facilement.

Ce que l'équipe m'a apporté

  • Les développeurs ont fourni un fichier SQL complet et testé (formula_one_fantasy_league.sql), ce qui m'a permis de configurer l'initialisation automatique de la base.
  • Ils ont identifié des besoins concrets (variables d'environnement pour la base, volumes pour le code en temps réel) que je n'avais pas anticipés au début.

Conclusion personnelle

Ce TP m'a montré l'importance d'une infrastructure bien pensée dans un projet d'équipe. Mon travail n'était pas visible directement dans l'interface utilisateur (pas de boutons, pas de design), mais sans lui, les développeurs n'auraient pas pu livrer leur code fonctionnel. La conteneurisation avec Docker et la mise en place d'un DNS interne sont des compétences que je réutiliserai dans tous mes futurs projets web, que ce soit en développement ou en production.

Ce que je retiens :

  • Un DNS interne simplifie considérablement la communication entre services.
  • Les healthchecks sont indispensables quand des services dépendent les uns des autres.
  • Bien documenter l'infrastructure permet à toute l'équipe d'être autonome.
  • Docker Compose est un outil puissant mais il faut comprendre chaque paramètre pour l'utiliser correctement.
