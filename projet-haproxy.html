1. Introduction et objectifs

Dans ce TP, nous devions construire une infrastructure complète imitant une architecture d'entreprise classique :

  • Une DMZ (zone démilitarisée) contenant les serveurs applicatifs.
  • Un pare-feu SNS qui sépare le réseau interne (Labo) de la DMZ et qui assure le NAT.
  • Un reverse-proxy HAProxy qui reçoit tout le trafic et le répartit entre plusieurs serveurs web.
  • Des backends (serveurs web) tournant sous Docker avec des IP fixes dans la DMZ.

Les objectifs pédagogiques étaient :

  1. Comprendre le routage et le NAT entre réseaux.
  2. Apprendre à utiliser Docker avec le driver macvlan pour donner des IP de la DMZ aux conteneurs.
  3. Maîtriser la configuration de HAProxy (HTTP, HTTPS, certificats wildcard vs dédiés, SNI).
  4. Mettre en œuvre des mécanismes d'équilibrage de charge (round-robin, leastconn) et de haute disponibilité (health checks, bascule automatique, maintenance).
  5. Sécuriser une VM Debian avec UFW.

Sur les backends

Une seule interface dans le réseau DMZ :

3. Installation des backends Docker dans la DMZ

3.1 Pourquoi Docker en macvlan ?

Nous voulions que les conteneurs aient des IP fixes dans la DMZ (10.0.0.x) afin que HAProxy puisse les joindre directement.
Le driver macvlan permet à un conteneur d'avoir sa propre interface réseau virtuelle avec une IP appartenant au réseau physique de l'hôte (ici la DMZ).

3.2 Installation de Docker

Sur chaque backend (10.0.0.100 et 10.0.0.101), nous avons installé Docker avec les commandes officielles :

apt update && apt install -y ca-certificates curl gnupg
install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/debian/gpg -o /etc/apt/keyrings/docker.asc
chmod a+r /etc/apt/keyrings/docker.asc
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/debian $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | tee /etc/apt/sources.list.d/docker.list > /dev/null
apt update
apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

Vérification : docker ps → affiche une table vide (OK).

3.3 Création du réseau macvlan

Sur un seul backend (nous avons choisi 10.0.0.100), nous avons créé le réseau Docker partagé :

docker network create -d macvlan \
  --subnet=10.0.0.0/24 \
  --gateway=10.0.0.1 \
  -o parent=enp0s8 \
  dmz_net
Note : enp0s8 est le nom de l'interface réseau qui porte l'IP 10.0.0.100 sur la VM. Nous avons vérifié avec ip a.

Observation : Le réseau dmz_net est maintenant visible sur les deux backends (car Docker propage les réseaux macvlan ? Non — en réalité, chaque backend doit avoir son propre réseau macvlan avec le même nom et la même configuration. Nous l'avons donc créé sur les deux VMs.)

3.4 Lancement des conteneurs Nginx

Backend 1 (10.0.0.100) :

docker run -d --name backend-1 --network dmz_net --ip 10.0.0.101 nginx

Backend 2 (10.0.0.101) :

docker run -d --name backend-2 --network dmz_net --ip 10.0.0.102 nginx
Pourquoi 10.0.0.101 sur le backend 1 ?
Le backend 1 a lui-même l'IP 10.0.0.100 (VM). Le conteneur ne doit pas prendre la même IP que l'hôte, donc nous lui avons donné 10.0.0.101.
De même, le backend 2 (VM 10.0.0.101) héberge un conteneur en 10.0.0.102.

3.5 Vérification des conteneurs

Depuis HaProxy (10.0.0.2) :

ping 10.0.0.101 # Réponses OK
ping 10.0.0.102 # Réponses OK
curl http://10.0.0.101 # Retourne la page d'accueil Nginx
curl http://10.0.0.102 # Idem

Conclusion : Les deux serveurs web sont opérationnels et accessibles dans la DMZ.

4. Configuration de HAProxy – passage de HTTP à HTTPS

4.1 Installation de HAProxy

Sur la VM HaProxy (172.25.1.2 / 10.0.0.2) :

sudo apt update
sudo apt install -y haproxy
haproxy -v # Affiche la version (ex: 2.6.x)

4.2 Première configuration : HTTP simple

Nous avons commencé par une configuration HTTP uniquement, pour valider le routage par nom d'hôte.

Fichier /etc/haproxy/haproxy.cfg :

defaults
  mode http
  option httplog
  timeout connect 5s
  timeout client 30s
  timeout server 30s

frontend fe_http
  bind :80
  acl host_node1 hdr(host) -i node1.example.test
  acl host_node2 hdr(host) -i node2.example.test
  use_backend be_node1 if host_node1
  use_backend be_node2 if host_node2
  default_backend be_node1

backend be_node1
  server n1 10.0.0.101:80 check

backend be_node2
  server n2 10.0.0.102:80 check

Explication :

  • acl (Access Control List) regarde l'en-tête Host de la requête HTTP.
  • Si le nom demandé est node1.example.test, on utilise be_node1 (qui pointe vers 10.0.0.101).
  • Par défaut, on envoie vers be_node1.

Test :

curl -H "Host: node1.example.test" http://172.25.1.2 # Réponse du conteneur backend-1
curl -H "Host: node2.example.test" http://172.25.1.2 # Réponse du conteneur backend-2
✅ Le routage par nom fonctionne.

4.3 Mise en place de HTTPS avec certificat wildcard

Nous avons voulu sécuriser les échanges avec TLS. Pour cela, nous avons créé notre propre Autorité de Certification (CA) interne.

Création de la CA

mkdir -p ~/ca && cd ~/ca
openssl genrsa -out myCA.key 4096
openssl req -x509 -new -nodes -sha256 -days 3650 \
  -key myCA.key \
  -subj "/C=FR/O=TP-HAProxy/OU=Student-CA/CN=My Student Lab CA" \
  -out myCA.crt

Le certificat myCA.crt a été installé dans le navigateur et dans le magasin de certificats du système d'exploitation du poste client (Windows, via certlm.msc).

Génération d'un certificat wildcard *.example.test

Un certificat wildcard couvre tous les sous-domaines (node1, node2, newsite, etc.).

openssl req -new -newkey rsa:2048 -nodes -keyout wildcard-example-test.key -subj "/CN=*.example.test" -out wildcard-example-test.csr
cat > san-wildcard.cnf < subjectAltName=DNS:*.example.test,DNS:example.test
EOF
openssl x509 -req -in wildcard-example-test.csr -CA myCA.crt -CAkey myCA.key -CAcreateserial -out wildcard-example-test.crt -days 365 -sha256 -extfile san-wildcard.cnf

Assemblage du fichier PEM pour HAProxy

HAProxy a besoin d'un fichier contenant le certificat, la clé privée, et éventuellement la CA.

sudo mkdir -p /etc/haproxy/certs
cat wildcard-example-test.crt wildcard-example-test.key myCA.crt | sudo tee /etc/haproxy/certs/wildcard-example-test.pem >/dev/null
sudo chown haproxy:haproxy /etc/haproxy/certs/wildcard-example-test.pem
sudo chmod 640 /etc/haproxy/certs/wildcard-example-test.pem

Configuration HAProxy avec redirection HTTP → HTTPS

frontend fe_https
  bind :80
  http-request redirect scheme https unless { ssl_fc }
  bind :443 ssl crt /etc/haproxy/certs/wildcard-example-test.pem

  acl host_node1 hdr(host) -i node1.example.test
  acl host_node2 hdr(host) -i node2.example.test
  use_backend be_node1 if host_node1
  use_backend be_node2 if host_node2
  default_backend be_node1

Ce qui change :

  • Le port 80 redirige automatiquement vers HTTPS.
  • Le port 443 écoute en SSL avec le certificat wildcard.

Test du wildcard :

curl -I https://newsite.example.test --resolve newsite.example.test:443:172.25.1.2
✅ Le serveur répond (même si aucun backend n'est configuré pour newsite, la connexion TLS réussit).

4.4 Comparaison avec des certificats dédiés

Nous avons ensuite généré un certificat pour node1.example.test et un pour node2.example.test, sans wildcard.

# Pour node1
openssl req -new -newkey rsa:2048 -nodes -keyout node1.key -subj "/CN=node1.example.test" -out node1.csr
echo "subjectAltName=DNS:node1.example.test" > san-node1.cnf
openssl x509 -req -in node1.csr -CA myCA.crt -CAkey myCA.key -CAcreateserial -out node1.crt -days 365 -sha256 -extfile san-node1.cnf

# Idem pour node2

Puis nous avons placé les deux fichiers PEM dans /etc/haproxy/certs/ et changé la directive bind :

bind :443 ssl crt /etc/haproxy/certs/

HAProxy utilise alors le SNI (Server Name Indication) : il regarde le nom demandé par le client pendant le handshake TLS et choisit le bon certificat.

Test :

curl --resolve node1.example.test:443:172.25.1.2 https://node1.example.test # ✅ OK
curl --resolve node2.example.test:443:172.25.1.2 https://node2.example.test # ✅ OK
curl --resolve newsite.example.test:443:172.25.1.2 https://newsite.example.test # ❌ Échec (pas de certificat)
✅ Wildcard
  • Un seul certificat pour tous les sous-domaines
  • Idéal pour labs / environnements de test
  • Nouveau sous-domaine = fonctionne immédiatement
  • ⚠️ Clé compromise = tous les sous-domaines compromis
🔒 Certificats dédiés
  • Un certificat par sous-domaine
  • Plus sûr (compartimentation)
  • Nouveau nom = nouveau certificat à générer
  • Maintenance plus lourde

Conclusion : Le wildcard est pratique pour les labs : un seul certificat couvre tous les sous-domaines. En production, on préfère des certificats dédiés pour limiter l'impact en cas de compromission d'une clé privée.

🔄 5. Équilibrage de charge & Haute Disponibilité

5.1 Backend unique avec deux serveurs

backend be_node1 balance roundrobin option httpchk GET / http-check expect rstatus 200 server n1 10.0.0.101:80 check inter 2s fall 2 rise 2 server n2 10.0.0.102:80 check inter 2s fall 2 rise 2

5.2 Mise en maintenance (MAINT)

echo "disable server be_node1/n2" | sudo socat stdio /run/haproxy/admin.sock # MAINT echo "enable server be_node1/n2" | sudo socat stdio /run/haproxy/admin.sock # Réactivation
