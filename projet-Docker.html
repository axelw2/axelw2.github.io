🐳 Infrastructure Docker – NTP, Syslog, TFTP

1. Introduction et objectifs

Dans ce TP, l'objectif était de simuler une petite infrastructure de services réseau, souvent présente dans un environnement d'entreprise ou de centre de données, puis d'y connecter un équipement Cisco (switch) pour qu'il consomme ces services.

Les services à déployer étaient :

  • NTP (Network Time Protocol) : pour synchroniser l'horloge du switch.
  • Syslog : pour centraliser les journaux d'événements du switch.
  • TFTP (Trivial File Transfer Protocol) : pour transférer des fichiers de configuration ou des images IOS.

Tous ces services devaient être conteneurisés avec Docker sur une machine Debian, puis consommés par un switch Cisco via le réseau local.

Objectifs pédagogiques :

  1. Maîtriser le déploiement de services réseau standards (NTP, Syslog, TFTP) via des conteneurs Docker prêts à l'emploi.
  2. Configurer finement ces services (serveur NTP privé, volume de données pour TFTP, agrégation de logs).
  3. Utiliser Docker Compose pour orchestrer plusieurs conteneurs.
  4. Connecter un équipement Cisco (switch) aux services Docker.
  5. Apprendre à configurer SSH, NTP, Syslog et TFTP sur un switch Cisco en mode ligne de commande.

2. Mise en place de l'infrastructure Docker

Toute l'infrastructure a été déployée sur une machine virtuelle Debian (peut aussi être une machine physique). Nous avons utilisé Docker et Docker Compose.

2.1 Récupération et configuration du serveur NTP

Nous avons utilisé le dépôt docker-ntp-server qui contient un serveur NTP basé sur chrony.

git clone https://github.com/manastria/docker-ntp-server.git cd docker-ntp-server/ntp

Le fichier de configuration chrony.conf a été modifié pour :

  • Commenter les serveurs NTP publics (pool.ntp.org).
  • Ajouter un serveur NTP privé (ici srv01.edgand.fr), qui servira de référence.
sudo nano chrony.conf

Contenu modifié :

#server 0.pool.ntp.org iburst #server 1.pool.ntp.org iburst #server 2.pool.ntp.org iburst #server 3.pool.ntp.org iburst server srv01.edgand.fr iburst
Remarque : En environnement réel, on utilise plusieurs serveurs publics. Ici, un seul serveur privé a été choisi pour l'exemple.

2.2 Déploiement du serveur de logs (Syslog)

Deuxième dépôt : docker_logs – un conteneur qui centralise les logs via Syslog et offre une interface web (LogAnalyzer).

git clone https://github.com/manastria/docker_logs.git cd docker_logs sudo docker-compose up -d

Une fois lancé, l'interface web est accessible sur http://localhost:8080.

Test d'envoi d'un message depuis la machine Debian :

logger -n localhost -P 514 -d "Ceci est un test" --priority user.info --tag test-local
  • -n localhost : serveur syslog (le conteneur)
  • -P 514 : port UDP standard
  • -d : mode UDP
  • --priority user.info : niveau de priorité
  • --tag test-local : étiquette du message
✅ Le message apparaît bien dans l'interface web.

2.3 Installation du serveur TFTP

Image Docker utilisée : pghalliday/tftp.

sudo docker pull pghalliday/tftp

Création d'un répertoire pour stocker les fichiers partagés :

sudo mkdir -p /sysadmin/tftp sudo chmod 777 /sysadmin/tftp # À restreindre en production
⚠️ Avertissement : Les droits 777 ne sont adaptés qu'à un laboratoire. En production, on utiliserait un utilisateur dédié et des droits plus stricts.

3. Orchestration multi-services avec Docker Compose

Pour gérer l'ensemble des trois services (NTP, logs, TFTP) de manière unifiée, nous avons créé un fichier /sysadmin/docker-compose.yml qui inclut les deux autres docker-compose.yml récupérés.

version: '3.8' services: tftp: image: pghalliday/tftp container_name: tftp volumes: - /home/sysadmin/tftpboot:/var/tftpboot # liaison avec le répertoire hôte network_mode: host restart: unless-stopped include: - ./docker-ntp-server/docker-compose.yml - ./docker_logs/docker-compose.yml
  • network_mode: host : le conteneur TFTP utilise directement le réseau de l'hôte (simplifie l'accès depuis le switch).
  • include : intègre les configurations des autres services.

Lancement global :

cd /sysadmin sudo docker-compose up -d sudo docker ps
✅ Les trois conteneurs sont actifs.

5. Connexion et configuration du switch Cisco

Le switch (modèle générique, par exemple un 2960) est connecté physiquement à la machine Debian par un câble Ethernet. Les deux sont sur le même sous-réseau.

5.1 Configuration IP et accès SSH

Adressage utilisé :

  • Machine Debian : 192.168.1.5 (NTP), 192.168.1.6 (Syslog), même IP pour TFTP
  • Switch : 192.168.1.2/24

Configuration du switch :

Switch> enable Switch# configure terminal Switch(config)# interface vlan 1 Switch(config-if)# ip address 192.168.1.2 255.255.255.0 Switch(config-if)# no shutdown Switch(config-if)# exit

Configuration SSH (indispensable pour une administration sécurisée) :

Switch(config)# hostname E1 E1(config)# ip domain-name edgand.fr E1(config)# crypto key generate rsa # clé RSA (taille 1024 ou 2048) E1(config)# username admin password toto E1(config)# ip ssh version 2 E1(config)# line vty 0 4 E1(config-line)# transport input ssh E1(config-line)# login local E1(config-line)# exit E1(config)# enable secret toto

Depuis Debian, tentative de connexion :

ssh admin@192.168.1.2
❌ Échec car le switch propose des algorithmes de chiffrement obsolètes (diffie-hellman-group1-sha1, aes128-cbc, etc.).

Solution : Créer une configuration spécifique pour le client SSH Debian.

~/.ssh/config :

Host cisco* KexAlgorithms +diffie-hellman-group1-sha1,diffie-hellman-group14-sha1 Ciphers aes128-cbc,3des-cbc,aes192-cbc,aes256-cbc HostKeyAlgorithms +ssh-rsa,ssh-dss Host cisco1 HostName 192.168.1.2 User admin

Puis :

ssh cisco1
✅ La connexion SSH fonctionne.

5.2 Synchronisation horaire (NTP)

Le switch doit obtenir l'heure depuis le serveur NTP Docker (192.168.1.5).

E1> enable E1# configure terminal E1(config)# ntp server 192.168.1.5 E1(config)# clock update-calendar E1(config)# clock timezone FR +1 E1(config)# clock summer-time FR recurring last SUN MAR 02:00 last SUN OCT 02:00 E1# show clock
✅ La sortie de show clock affiche l'heure correcte, avec le fuseau horaire français (UTC+1) et l'heure d'été automatique.
Vérification : show ntp status donne l'état de la synchronisation. Si l'heure ne se met pas à jour, vérifier que le serveur NTP Docker répond bien (ntpq -p depuis la Debian).

5.3 Centralisation des logs (Syslog)

Le switch envoie ses journaux au serveur Syslog Docker (192.168.1.6).

E1(config)# logging host 192.168.1.6 E1(config)# logging trap informational # niveau de sévérité minimal E1# show logging

Envoi d'un message de test depuis le switch :

E1# send log "Système en arrêt imminent" level emergencies
✅ Sur l'interface web http://localhost:8080 (depuis la machine Debian), le message apparaît dans LogAnalyzer.

5.4 Transfert de fichiers (TFTP)

Le serveur TFTP tourne sur 192.168.1.5 (même hôte que NTP). Depuis le switch, on peut sauvegarder la configuration startup-config sur le serveur TFTP :

E1# copy startup-config tftp: Address or name of remote host []? 192.168.1.5 Destination filename [E1-confg]? config-switch.txt
✅ Le fichier config-switch.txt est créé dans /home/sysadmin/tftpboot sur la machine Debian.

Restaurer une configuration depuis TFTP :

E1# copy tftp: startup-config
✅ Le transfert fonctionne.

6. Problèmes rencontrés et solutions

ProblèmeCauseSolution
SSH depuis Debian vers switch Switch utilise des algorithmes obsolètes (diffie-hellman-group1-sha1, aes128-cbc) Modifier le client SSH via ~/.ssh/config pour accepter ces algorithmes.
NTP ne synchronise pas Adresse du serveur NTP incorrecte ou pare-feu bloquant UDP 123 Vérifier que la Debian a bien l'IP 192.168.1.5 et que Docker expose bien le port NTP.
Les logs n'apparaissent pas dans LogAnalyzer Le switch envoie en TCP alors que le conteneur écoute en UDP (ou l'inverse) logging host 192.168.1.6 utilise UDP par défaut ; s'assurer que docker_logs accepte l'UDP.
TFTP : permission denied Droits du répertoire /home/sysadmin/tftpboot trop restrictifs pour le conteneur sudo chmod 777 /home/sysadmin/tftpboot (en lab) ou ajuster l'UID du conteneur.
⚠️ Remarque importante : Les adresses IP des services (NTP, Syslog, TFTP) étaient toutes différentes dans le tutoriel initial (192.168.1.5 pour NTP, 192.168.1.6 pour logs). En pratique, un seul serveur Docker peut héberger les trois services sur la même IP (par exemple 192.168.1.100). Nous avons choisi de simplifier en utilisant la même IP pour les trois, mais il faut alors adapter les noms de conteneurs ou utiliser des ports différents (ce qui est moins pratique).

7. Conclusion

Ce TP a permis de mettre en place une infrastructure complète de services réseau (NTP, TFTP, Syslog) conteneurisée avec Docker, puis de connecter un switch Cisco pour qu'il utilise ces services.

Points clés réalisés :

  1. Déploiement conteneurisé : Utilisation d'images Docker existantes (pghalliday/tftp, manastria/docker-ntp-server, docker_logs).
  2. Orchestration : Un seul docker-compose.yml intègre les trois services (via include et network_mode: host).
  3. Configuration Cisco :
    • SSH sur le switch avec adaptation du client pour les anciens algorithmes.
    • Synchronisation horaire via NTP avec fuseau horaire français et heure d'été.
    • Centralisation des logs Syslog vers l'interface web LogAnalyzer.
    • Transfert de fichiers (sauvegarde/restauration de config) via TFTP.
  4. Tests complets : Vérification de chaque service depuis le switch et depuis la machine hôte.

Apprentissages clés :

  • La conteneurisation simplifie le déploiement de services réseau standards.
  • Docker Compose permet d'orchestrer des services hétérogènes (NTP, TFTP, logs) facilement.
  • Les équipements Cisco anciens (ou en laboratoire) nécessitent parfois d'assouplir la configuration du client SSH moderne.
  • La centralisation des logs est un besoin critique en administration système et réseau.
  • TFTP reste très utile pour la sauvegarde/restauration de configurations ou le transfert d'images IOS, malgré son absence de sécurité.
