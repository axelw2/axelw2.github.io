1. Introduction et objectifs

Dans un réseau local, les liaisons entre switches ou entre un switch et un serveur peuvent rapidement devenir saturées. L'ajout d'un second câble Ethernet ne résout pas le problème : Spanning Tree Protocol (STP) bloquerait l'un des deux liens pour éviter une boucle.

L'agrégation de liens (link aggregation) permet de contourner cette limitation en regroupant plusieurs liens physiques en un seul lien logique. Le trafic est réparti entre les différents liens, augmentant ainsi la bande passante disponible tout en assurant la tolérance aux pannes.

Objectifs du TP :

  1. Comprendre le principe de l'agrégation de liens et ses avantages.
  2. Différencier les technologies : EtherChannel (propriétaire Cisco) vs 802.3ad/LACP (standard ouvert).
  3. Identifier les conditions nécessaires pour qu'une agrégation fonctionne.
  4. Configurer un EtherChannel en mode LACP entre deux switches Cisco.
  5. Vérifier le bon fonctionnement et l'intégration avec STP.
  6. Appliquer des configurations avancées (filtrage VLAN, VLAN natif).

2. Qu'est-ce que l'agrégation de liens ?

L'agrégation de liens (Link Aggregation) est une technique qui permet de combiner plusieurs interfaces réseau physiques en une seule interface logique. Cette interface logique est appelée :

  • EtherChannel chez Cisco
  • Port Channel ou LAG (Link Aggregation Group) dans le langage générique
  • Bonding ou Teaming côté serveur
CaractéristiqueDescription
Nombre de liensDe 2 à 8 liens actifs (parfois plus selon les équipements)\/th>
Bande passante Addition des débits de chaque lien (ex : 2 × 1 Gbps = 2 Gbps théoriques)<\/strong>
Tolérance aux pannes Si un lien tombe, le trafic bascule sur les autres liens restants<\/strong>
Répartition de charge Les paquets sont répartis entre les liens selon divers critères (MAC, IP, port)<\/strong>
Vues par STP L'agrégat est vu comme un seul lien logique → pas de boucle<\/strong>

3. Pourquoi agréger des liens ?

Avantages :

AvantageExplication
Augmentation de la bande passante Un flux unique ne dépasse pas la capacité d'un seul lien, mais plusieurs flux peuvent être répartis. Exemple : 2 serveurs communiquant avec 2 clients différents peuvent utiliser chacun un lien différent.
Tolérance aux pannes Un lien physique peut tomber sans interruption de service (le trafic continue sur les autres liens).
Simplicité logique L'administrateur configure une seule interface logique (port-channel) au lieu de plusieurs interfaces physiques.
Économie Plutôt que de passer à un standard supérieur (ex : 10 Gbps), on agrège plusieurs liens 1 Gbps existants.

Utilisations typiques :

  • Dorsale du réseau local : entre switches de distribution et de cœur.
  • Accès serveurs : entre un switch et un serveur équipé de plusieurs cartes réseau (NIC teaming).
  • Interconnexion de sites : entre routeurs ou sur des liaisons louées.

4. Les deux grandes familles : EtherChannel vs 802.3ad

4.1. EtherChannel (Cisco)

EtherChannel est la technologie propriétaire développée par Cisco. Elle repose historiquement sur le protocole PAgP (Port Aggregation Protocol).

AspectDescription
Propriétaire Cisco uniquement
Protocole de négociation PAgP (Cisco) ou LACP (supporté aussi)
Modes PAgP desirable (négocie activement), auto (attend la négociation)
Répartition de charge Plusieurs algorithmes configurables (MAC, IP, TCP/UDP ports)

4.2. 802.3ad / LACP (standard IEEE)

En 2000, l'IEEE a standardisé une version ouverte de l'agrégation de liens : la norme 802.3ad, également appelée LACP (Link Aggregation Control Protocol).

AspectDescription
Standard IEEE 802.3ad (aujourd'hui intégré dans IEEE 802.1AX)
Interopérabilité Fonctionne entre équipements de fabricants différents
Protocole LACP uniquement
Modes LACP active (négocie activement), passive (attend la négociation)

4.3. Tableau comparatif

AspectEtherChannel (Cisco)802.3ad / LACP (IEEE)
Nature Propriétaire Cisco Standard ouvert
Protocole de négociation PAgP (ou LACP) LACP uniquement
Compatibilité matérielle Cisco uniquement Multi-fabricants
Modes disponibles desirable, auto, on active, passive
Popularité Très répandu dans les environnements Cisco Standard dans les environnements hétérogènes
💡 Remarque : Les switches Cisco modernes supportent à la fois PAgP et LACP. Dans ce TP, nous utiliserons LACP pour sa compatibilité universelle.

5. Conditions préalables à l'agrégation

Pour qu'une agrégation de liens fonctionne correctement, tous les ports physiques du groupe doivent respecter strictement les mêmes paramètres.

ConditionExigencePourquoi ?
Vitesse Identique sur tous les ports (ex: 1 Gbps) Un déséquilibre rendrait l'agrégation incohérente
Duplex Full-duplex sur tous les ports Le half-duplex n'est pas compatible avec l'agrégation moderne
VLANs (mode trunk) Même liste de VLANs autorisés Sinon, certains paquets seraient perdus
VLAN natif Identique sur tous les ports Doit correspondre entre les deux extrémités
Type de liaison Tous les ports en accès ou tous en trunk Mélange non supporté
Support matériel Interfaces compatibles EtherChannel Tous les ports ne le sont pas (selon les modèles)
⚠️ Conséquence : Si ces conditions ne sont pas remplies, le protocole de négociation (PAgP ou LACP) détecte l'incohérence et refuse d'établir l'agrégation.

6. Interaction avec les autres protocoles

6.1. Spanning Tree Protocol (STP)

STP a pour rôle d'éviter les boucles réseau en bloquant certains ports. Sans agrégation, deux liens entre deux switches verraient l'un des deux ports bloqué.

Avec EtherChannel :

  • STP voit l'ensemble du groupe comme un seul lien logique (le Port-Channel).
  • Les BPDU (messages STP) sont envoyés sur un seul des liens physiques du groupe.
  • Aucun port physique n'est bloqué individuellement ; c'est le Port-Channel tout entier qui est soit en forwarding, soit en blocking.

6.2. VLANs et trunk

EtherChannel supporte le transport de plusieurs VLANs via la configuration en mode trunk (802.1Q ou ISL).

Règles importantes :

  • Si un seul port du groupe est configuré en trunk, tous les ports du groupe héritent de cette configuration.
  • Le VLAN natif doit être identique sur tous les ports du groupe.
  • La liste des VLANs autorisés doit être cohérente sur tous les ports.

7. Mise en pratique : configuration LACP entre deux switches Cisco

7.1. Topologie et contexte

SW1 ────────────── SW2
Gi0/1 ────────── Gi0/1
Gi0/2 ────────── Gi0/2

Nous disposons de deux switches Cisco (SW1 et SW2) reliés par deux câbles Gigabit Ethernet :

  • GigabitEthernet0/1 (SW1) ←→ GigabitEthernet0/1 (SW2)
  • GigabitEthernet0/2 (SW1) ←→ GigabitEthernet0/2 (SW2)
⚠️ Problème initial : Avec STP actif par défaut, l'un des deux ports est bloqué (par exemple, Gi0/2 sur SW2).

7.2. Vérification initiale de STP

Sur SW1, vérifions l'état de STP :

SW1> enable SW1# show spanning-tree
VLAN0001 Spanning tree enabled protocol ieee Root ID Priority 32769 Address 0001.1234.5678 Cost 4 Port Gi0/1 (Forwarding) Bridge ID Priority 32769 Address 0001.8765.4321 Aging Time 300 Interface Role State Gi0/1 Root Forwarding Gi0/2 Altn Blocking <-- un des deux liens est bloqué
❌ On observe bien qu'un seul lien est en état Forwarding. L'autre est bloqué pour éviter une boucle.

7.3. Configuration du EtherChannel en mode LACP

Sur SW1 (mode actif) :

SW1> enable SW1# configure terminal SW1(config)# interface range GigabitEthernet0/1 - 2 SW1(config-if-range)# channel-group 1 mode active SW1(config-if-range)# exit SW1(config)# interface port-channel 1 SW1(config-if)# switchport mode trunk SW1(config-if)# exit SW1(config)# exit SW1# write memory

Sur SW2 (mode passif ou actif) :

SW2> enable SW2# configure terminal SW2(config)# interface range GigabitEthernet0/1 - 2 SW2(config-if-range)# channel-group 1 mode passive SW2(config-if-range)# exit SW2(config)# interface port-channel 1 SW2(config-if)# switchport mode trunk SW2(config-if)# exit SW2(config)# exit SW2# write memory
Mode LACPComportement
active Le port négocie activement l'agrégation en envoyant des trames LACP
passive Le port répond aux trames LACP reçues mais n'initie pas la négociation
on Agrégation forcée (sans négociation) – à éviter car ne détecte pas les erreurs
💡 Recommandation : Utiliser active sur les deux extrémités pour un démarrage plus rapide et symétrique.

7.4. Vérification de l'agrégation

Commande principale :

SW1# show etherchannel summary
Flags: D - down P - bundled in port-channel I - stand-alone s - suspended H - Hot-standby (LACP only) R - Layer3 S - Layer2 U - in use f - failed to allocate aggregator Number of channel-groups in use: 1 Number of aggregators: 1 Group Port-channel Protocol Ports ------+-------------+-----------+----------------------------------------------- 1 Po1(SU) LACP Gi0/1(P) Gi0/2(P)
✅ Les deux ports sont "bundled" (agrégés) et actifs.

Autres commandes utiles :

# Afficher les détails du Port-Channel SW1# show interface port-channel 1 # Afficher l'état de STP sur le Port-Channel (doit être un seul lien) SW1# show spanning-tree | include Po1 # Afficher la répartition de charge SW1# show etherchannel load-balance

8. Configuration avancée : filtrage VLAN et VLAN natif

Une fois l'EtherChannel opérationnel, on peut affiner la configuration du trunk.

8.1. Limiter les VLANs autorisés

SW1(config)# interface port-channel 1 SW1(config-if)# switchport trunk allowed vlan 50,60 SW1(config-if)# exit
💡 Avant : tous les VLANs (1 à 4094) sont autorisés.
💡 Après : seuls les VLANs 50 et 60 peuvent transiter.

8.2. Définir un VLAN natif différent du VLAN 1

SW1(config)# interface port-channel 1 SW1(config-if)# switchport trunk native vlan 99 SW1(config-if)# exit
⚠️ Important : Le VLAN natif doit être identique sur les deux extrémités du trunk, sinon des incohérences apparaissent (mauvais routage des trames non taguées).

9. Problèmes rencontrés et solutions

ProblèmeCauseSolution
L'EtherChannel ne se forme pas Les modes LACP sont incompatibles (ex: passive des deux côtés) Mettre au moins un côté en active
Les ports restent en mode stand-alone (I) Les paramètres (vitesse, duplex, VLAN) ne sont pas identiques Vérifier show interfaces status. Ajuster la liste des VLANs autorisés
STP bloque encore un port L'EtherChannel n'est pas correctement formé Refaire la configuration, vérifier show etherchannel summary
Perte de trafic après agrégation Répartition de charge inégale Changer l'algorithme : port-channel load-balance src-dst-ip
Le trunk ne passe pas les VLANs VLAN natif différent ou VLANs non autorisés Vérifier show interfaces trunk. Appliquer switchport trunk allowed vlan add
LACP ne se négocie pas Un port est en mode desirable (PAgP) Utiliser channel-group 1 mode active

10. Synthèse des commandes

show interfaces trunk
CommandeRôle
interface range GigabitEthernet0/1 - 2 Sélectionne une plage d'interfaces
channel-group 1 mode active Crée un EtherChannel groupe 1 en mode LACP actif
interface port-channel 1 Entre dans la configuration de l'interface logique
switchport mode trunk Configure le Port-Channel en mode trunk
switchport trunk allowed vlan 50,60 Limite les VLANs autorisés sur le trunk
switchport trunk native vlan 99 Définit le VLAN natif
show etherchannel summary Affiche l'état de tous les EtherChannels
show spanning-tree Vérifie que STP voit le Port-Channel comme un lien unique
Vérifie la configuration trunk du Port-Channel

11. Conclusion

Ce TP nous a permis de maîtriser un concept fondamental de l'infrastructure réseau : l'agrégation de liens.

Ce que nous avons appris :

  1. Pourquoi agréger des liens : augmentation de bande passante, tolérance aux pannes, simplification administrative.
  2. Les deux technologies principales : EtherChannel/PAgP (Cisco) et 802.3ad/LACP (standard IEEE).
  3. Les conditions de succès : vitesse, duplex, VLANs identiques sur tous les ports.
  4. La configuration Cisco (LACP) : channel-group 1 mode active puis configuration du Port-Channel.
  5. La vérification : show etherchannel summary pour confirmer l'agrégation.
À retenirPourquoi c'est important
LACP nécessite au moins un côté en active Sinon, aucune négociation n'est initiée
Tous les ports du groupe doivent avoir les mêmes VLANs autorisés Un port avec une liste différente serait exclu du groupe
STP ne bloque plus les ports individuels C'est tout l'EtherChannel qui est soit forwarding, soit blocking
Un EtherChannel peut être en mode trunk Permet de transporter plusieurs VLANs sur l'agrégat
La répartition de charge n'est pas parfaite pour un seul flux Un seul flux utilise toujours un seul lien physique

Application professionnelle :

Cette configuration est omniprésente dans les datacenters et les réseaux d'entreprise :

  • Entre switches de distribution : agrégation de 4 liens 10 Gbps vers le cœur de réseau.
  • Vers des serveurs virtualisés : un serveur ESXi ou Hyper-V avec plusieurs NICs en teaming LACP.
  • Interconnexion géographique : agrégation de liaisons louées pour augmenter le débit entre deux sites distants.
