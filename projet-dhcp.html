1. Introduction et objectifs

Le DHCP (Dynamic Host Configuration Protocol) est un protocole réseau essentiel qui permet d'attribuer automatiquement des adresses IP aux ordinateurs clients. Sans DHCP, chaque poste doit être configuré manuellement (adresse IP, masque, passerelle, DNS), ce qui devient rapidement ingérable sur un parc de plusieurs dizaines ou centaines de machines.

Objectifs du TP :

  1. Comprendre le principe du DHCP : bail (lease), plage d'attribution (range), durée de validité.
  2. Installer et configurer un serveur DHCP sur Debian avec le paquet isc-dhcp-server.
  3. Configurer une interface réseau en statique sur le serveur (prérequis obligatoire).
  4. Tester l'attribution automatique depuis un client Windows ou Linux.
  5. Valider la connectivité entre le client et le serveur.

2. Architecture et matériel utilisé

L'infrastructure se compose de trois éléments physiques/logiques :

ÉlémentRôleConfiguration réseau
PC physique (hôte)Héberge la VM serveurConnecté au switch (port Fa0/1)
VM Debian (serveur DHCP)Distribue les adresses IPIP statique 192.168.1.10 sur interface en mode Bridge
Switch (non configuré)Connecte serveur et clientDefault : tous les ports en VLAN 1
PC clientReçoit une IP automatiquementConnecté au switch (port Fa0/2)
💡 Remarque importante : Le serveur DHCP doit impérativement avoir une adresse IP statique. En effet, si le serveur lui-même devait obtenir son IP via DHCP, cela créerait une dépendance circulaire (l'œuf et la poule).

Schéma de connexion :

PC physique (hôte VirtualBox)
│
├── VM Debian (serveur DHCP) ──┐
│ │
└── (Bridge) ──────────────────┼── switch ── PC client
│
(ports Fa0/1 et Fa0/2)

3. Configuration de la machine virtuelle serveur

3.1 Création de la VM

Sous VirtualBox, nous avons créé une machine virtuelle Debian 12 avec les caractéristiques suivantes :

  • Système : Debian 12 (bookworm) - netinstall
  • Mémoire : 1 Go (suffisant pour un serveur DHCP)
  • Disque : 10 Go
  • Réseau : carte en mode Bridge (permet à la VM d'être sur le même réseau physique que l'hôte)

3.2 Configuration de l'adresse IP statique

Une fois Debian installée, nous avons configuré l'interface réseau en statique. Sur Debian 12, la configuration se fait via le fichier /etc/network/interfaces ou via nmcli.

Méthode retenue (fichier interfaces) :

sudo nano /etc/network/interfaces

Contenu :

auto eth0 iface eth0 inet static address 192.168.1.10 netmask 255.255.255.0 gateway 192.168.1.1 (optionnel pour la phase de test)

Puis redémarrage du service réseau :

sudo systemctl restart networking

Vérification :

ip addr show eth0
💡 Note : Le nom de l'interface peut être enp0s3 ou ens33 selon les versions. Dans ce TP, nous utiliserons eth0 par convention.

4. Installation et configuration du serveur DHCP

4.1. Installation du paquet isc-dhcp-server

isc-dhcp-server est le serveur DHCP historique et le plus répandu sur les distributions Debian/Ubuntu.

sudo apt update sudo apt install isc-dhcp-server -y

Vérification de l'installation :

dpkg -l | grep isc-dhcp-server

4.2. Configuration du fichier dhcpd.conf

Le fichier de configuration principal est /etc/dhcp/dhcpd.conf. Nous l'avons édité :

sudo nano /etc/dhcp/dhcpd.conf

Configuration appliquée :

# Paramètres globaux (par défaut pour tous les sous-réseaux) # Nom de domaine local option domain-name "security-helpzone.lan"; # Serveurs DNS (ici : serveur local + Google) option domain-name-servers 192.168.1.2, 8.8.8.8; # Durée du bail (lease time) default-lease-time 86400; # 24 heures max-lease-time 604800; # 1 semaine # Ce serveur est autoritaire (pour les réseaux qu'il gère) authoritative; # Définition du sous-réseau à gérer subnet 192.168.1.0 netmask 255.255.255.0 { # Plage d'adresses à distribuer range 192.168.1.100 192.168.1.200; # Masque de sous-réseau (optionnel ici car déjà dans la directive subnet) option subnet-mask 255.255.255.0; # Passerelle par défaut (routeur) option routers 192.168.1.1; # Serveurs DNS (spécifiques à ce sous-réseau, surcharge les globaux si présents) option domain-name-servers 8.8.8.8, 8.8.4.4; }

Explication des directives principales :

DirectiveRôle
option domain-nameNom de domaine (suffixe DNS)
option domain-name-serversAdresses des serveurs DNS
default-lease-timeDurée du bail par défaut (en secondes)
max-lease-timeDurée maximale du bail
authoritativeLe serveur se déclare autoritaire pour le réseau (répond même si l'adresse semble fausse)
subnet ... netmask ...Déclare le sous-réseau géré
rangePlage d'adresses à distribuer (début et fin)
option routersPasserelle par défaut
option subnet-maskMasque de sous-réseau

4.3. Vérification de la syntaxe

Avant de démarrer le service, il est impératif de vérifier la configuration :

sudo /sbin/dhcpd -t
✅ Sortie attendue : Configuration file syntax check successful.
⚠️ En cas d'erreur : le message indique la ligne problématique. Corrigez puis relancez la vérification.

4.4. Spécification de l'interface d'écoute

Par défaut, le serveur DHCP écoute sur toutes les interfaces. Nous devons lui indiquer sur quelle(s) interface(s) il doit distribuer des IP.

Fichier : /etc/default/isc-dhcp-server

sudo nano /etc/default/isc-dhcp-server

Modification :

INTERFACESv4="eth0"
💡 Note : Si votre interface s'appelle enp0s3, remplacez eth0 par le bon nom (vérifiez avec ip a).

4.5. Démarrage et activation du service

sudo systemctl restart isc-dhcp-server sudo systemctl enable isc-dhcp-server # démarrage automatique au boot sudo systemctl status isc-dhcp-server
✅ Le statut doit indiquer active (running) et enabled.

5. Tests de validation

5.1. Vérification des logs

sudo tail -f /var/log/syslog | grep dhcp

Ou avec journalctl :

sudo journalctl -u isc-dhcp-server -f
✅ Aucune erreur ne doit apparaître. En cas de problème, les logs sont le premier outil de diagnostic.

5.2. Côté client Windows

Nous avons connecté un PC client (Windows) sur le même switch, port Fa0/2.

Libération de l'ancienne IP (optionnel) :

ipconfig /release

Renouvellement de l'adresse :

ipconfig /renew

Vérification :

ipconfig /all

Nous avons observé :

  • Adresse IPv4 : 192.168.1.100 (première IP de la plage)
  • Masque : 255.255.255.0
  • Passerelle : 192.168.1.1
  • DNS : 8.8.8.8
✅ Le client a bien reçu une IP de la plage configurée.

5.3. Côté client Linux

Pour un client Linux (Debian/Ubuntu) :

sudo dhclient -r eth0 # libère l'IP existante sudo dhclient eth0 # demande une nouvelle IP ip a # affiche l'IP obtenue

Ou plus simplement :

sudo dhclient -v

5.4. Tests de connectivité

Depuis le client, ping vers le serveur DHCP :

ping 192.168.1.10

Depuis le serveur, ping vers le client :

ping 192.168.1.100
✅ Résultat attendu : Reply from ... (réponses reçues dans les deux sens).

5.5. Consultation des baux DHCP

Sur le serveur, les baux (adresses attribuées) sont stockés dans :

sudo cat /var/lib/dhcp/dhcpd.leases

On y trouve :

  • L'adresse IP attribuée
  • L'adresse MAC du client
  • La date de début et de fin du bail

6. Extraits de configuration

Configuration minimale fonctionnelle

Pour un cas simple, voici la configuration minimale qui fonctionne :

subnet 192.168.1.0 netmask 255.255.255.0 { range 192.168.1.100 192.168.1.200; option routers 192.168.1.1; option domain-name-servers 8.8.8.8; }
✅ Avec cette configuration, les clients obtiendront une IP dans la plage, la passerelle sera 192.168.1.1 et le DNS 8.8.8.8.

Configuration avec réservation statique (optionnelle)

Pour qu'une machine spécifique (identifiée par son adresse MAC) reçoive toujours la même IP :

host client-fixe { hardware ethernet 00:11:22:33:44:55; fixed-address 192.168.1.150; }
💡 Note : Cette section se place à l'intérieur du bloc subnet.

7. Conclusion

Ce TP nous a permis de mettre en place un service DHCP complet et fonctionnel sur une machine virtuelle Debian.

Ce que nous avons appris :

  1. Le DHCP est un protocole broadcast : le client envoie une requête DHCPDISCOVER à destination de l'adresse 255.255.255.255. Le serveur répond avec DHCPOFFER, puis le client termine par DHCPREQUEST et DHCPACK.
  2. Le serveur DHCP doit avoir une IP statique : sinon, il ne peut pas fonctionner (dépendance circulaire).
  3. La syntaxe de dhcpd.conf : les directives globales, les blocs subnet, les options spécifiques.
  4. Les logs sont essentiels : /var/log/syslog ou journalctl sont les premiers outils de diagnostic.
  5. Le mode Bridge de VirtualBox : permet à la VM d'être sur le même réseau que l'hôte physique.

Points de vigilance pour la production :

  • Ne jamais configurer deux serveurs DHCP sur le même sous-réseau (risque de conflits).
  • Toujours configurer authoritative pour un serveur principal (le serveur répondra même si l'adresse demandée semble hors plage, forçant le client à accepter la proposition).
  • Réserver les adresses des serveurs (imprimantes, serveurs DHCP, routeurs) en dehors de la plage dynamique. La plage 192.168.1.100-200 est bien choisie, les adresses 192.168.1.1 à 192.168.1.99 sont réservées pour les équipements fixes.
