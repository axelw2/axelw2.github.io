1. Introduction et objectifs

Ce TP avait pour but de nous familiariser avec PowerShell, l'environnement de ligne de commande et de scripting de Windows. Nous avons abordé trois grands domaines :

  1. La manipulation de fichiers et dossiers : création, copie, déplacement, suppression.
  2. La gestion des comptes utilisateurs locaux via le fournisseur ADSI (Active Directory Services Interface).
  3. L'interrogation du système via WMI (Windows Management Instrumentation) et CIM.

L'objectif pédagogique était d'acquérir les compétences de base pour automatiser les tâches courantes d'administration système sous Windows, en préparation de scripts plus complexes en entreprise.

2. Prise en main de PowerShell

PowerShell est un shell orienté objet : chaque commande retourne des objets, pas du texte brut. Cela permet d'accéder facilement aux propriétés des objets retournés.

2.1. Navigation dans l'arborescence

Avant toute manipulation, nous avons appris à nous déplacer dans l'arborescence du système de fichiers.

ActionCommandeAlias
Afficher le dossier courantGet-Locationpwd
Se déplacer dans un dossierSet-Location C:\testPowerShellcd
Remonter d'un niveauSet-Location ..cd ..
Afficher la liste des fichiers/dossiersGet-ChildItemgci ou ls
💡 Explication : PowerShell utilise des verbes (Get, Set, Remove, etc.) suivis d'un nom (Location, ChildItem, etc.). Les alias (comme cd ou ls) permettent de conserver les habitudes des anciens shells.

Exemple concret : Création d'un dossier de travail et déplacement dedans.

New-Item -Path .\testPowerShell -ItemType Directory cd .\testPowerShell Get-Location
✅ Le dossier testPowerShell est créé et devient le répertoire courant.

2.2. Création et manipulation de fichiers/dossiers

Nous avons appris à créer des fichiers avec du contenu, à les copier et à afficher leurs propriétés.

Création d'un fichier texte :

Set-Content -Path C:\testPowerShell\test1.txt -Value "Tp PowerShell 1"

Vérification du contenu :

Get-Content C:\testPowerShell\test1.txt

Affichage des propriétés d'un fichier stocké dans une variable :

$fichier = Get-ChildItem C:\testPowerShell\test1.txt Write-Host "Nom du fichier: $($fichier.Name)" Write-Host "Taille du fichier (en octets): $($fichier.Length)" Write-Host "Extension du fichier: $($fichier.Extension)" Write-Host "Date du dernier accès: $($fichier.LastAccessTime)"
💡 Point clé : La syntaxe $($fichier.Name) permet d'afficher la valeur d'une propriété d'un objet à l'intérieur d'une chaîne de caractères.

Copie d'un fichier :

$fichier.CopyTo("C:\testPowerShell\essai2.txt") # Vérification Test-Path "C:\testPowerShell\essai2.txt" # Retourne True

2.3. Test d'existence et suppression

Tester si un dossier existe :

Test-Path C:\Windows # Retourne True

Supprimer un dossier avec tout son contenu (attention à la puissance de cette commande !) :

Remove-Item -Path .\test2dossier -Recurse -Force
💡 Explication : -Recurse supprime les sous-dossiers, -Force supprime sans demande de confirmation. En production, il est conseillé d'utiliser -WhatIf d'abord pour simuler la suppression.

Affectation du résultat d'une commande à une variable :

$loc = Get-Location $loc.Path # Affiche le chemin $loc | Get-Member # Affiche toutes les propriétés et méthodes disponibles
✅ Get-Member est une commande essentielle pour découvrir ce qu'un objet peut faire.

3. Gestion des comptes locaux avec ADSI

ADSI (Active Directory Services Interface) permet d'interagir avec l'annuaire Windows. Ici, nous l'avons utilisé pour gérer les comptes utilisateurs locaux (base SAM).

💡 Pourquoi ADSI et pas une autre méthode ? Les cmdlets *-LocalUser (New-LocalUser, Get-LocalUser, etc.) n'existent que sur Windows 10/11 et Windows Server 2016+. ADSI fonctionne sur toutes les versions de Windows, y compris les plus anciennes.

3.1. Lecture des informations d'un compte

Le script lastlogin.ps1 interroge un compte local et affiche sa date de dernière connexion, sa description et son nom complet.

$nom = Read-Host -Prompt "Saisir un nom de compte local" $compte = [ADSI]"WinNT://./$nom" if ($compte.Path) { Write-Host $compte.LastLogin Write-Host $compte.Description Write-Host $compte.FullName } else { Write-Host "$nom non trouvé" }
💡 Explication détaillée :
  • [ADSI]"WinNT://./$nom" : se connecte à la base d'utilisateurs locale (SAM) et récupère l'objet utilisateur.
  • Le ./ signifie "machine locale". Pour un serveur distant, on utiliserait "WinNT://serveur/nom".
  • Si l'utilisateur existe, $compte.Path n'est pas vide. On peut alors accéder aux propriétés LastLogin, Description, FullName.

3.2. Création interactive d'un compte

Le script ajoutCompte.ps1 crée un nouvel utilisateur local avec saisie interactive :

$local = [ADSI]"WinNT://." $nom = Read-Host -Prompt "Saisir un nom" $description = Read-Host -Prompt "Saisir une description" $FullName = Read-Host -Prompt "Saisir un nom complet" $compte = [ADSI]"WinNT://./$nom" if (!$compte.path) { $utilisateur = $local.create("user", $nom) $utilisateur.InvokeSet("Description", $description) $utilisateur.InvokeSet("FullName", $FullName) $utilisateur.CommitChanges() Write-Host "$nom ajouté" } else { Write-Host "$nom existe déjà" }
💡 Explication détaillée :
  • $local = [ADSI]"WinNT://." : se connecte à la base SAM locale (le point représente la machine locale).
  • $local.create("user", $nom) : crée l'objet utilisateur en mémoire.
  • InvokeSet : méthode pour définir des propriétés qui ne sont pas exposées directement en PowerShell.
  • CommitChanges() : valide la création dans la base SAM. Sans cette ligne, l'utilisateur n'est pas réellement créé.

3.3. Création massive depuis un fichier texte

Le fichier listeCompte.txt doit avoir la structure suivante (séparateur /) :

nomUtilisateur/nomComplet/description dupont/Jean Dupont/Comptable martin/Sophie Martin/Commerciale

Le script ajout utilisateur via txt.ps1 lit ce fichier et crée tous les comptes :

$fichier = "C:\testPowerShell\listeCompte.txt" $local = [ADSI]"WinNT://." if (Test-Path $fichier) { $colLignes = Get-Content $fichier foreach ($ligne in $colLignes) { $tabCompte = $ligne.Split("/") $nom = $tabCompte[0] $FullName = $tabCompte[1] $description = $tabCompte[2] $compte = [ADSI]"WinNT://./$nom" if (!$compte.path) { $utilisateur = $local.create("user", $nom) $utilisateur.InvokeSet("Description", $description) $utilisateur.InvokeSet("FullName", $FullName) $utilisateur.CommitChanges() Write-Host "$nom ajouté" } else { Write-Host "$nom existe déjà" } } }
💡 Explication détaillée :
  • Test-Path $fichier : vérifie que le fichier existe avant de tenter de le lire.
  • Get-Content $fichier : lit toutes les lignes du fichier dans un tableau.
  • foreach ($ligne in $colLignes) : parcourt chaque ligne.
  • Split("/") : découpe la ligne en trois parties (nom, nom complet, description).

3.4. Suppression massive depuis un fichier texte

Le script suppression utilisateur via txt.ps1 utilise le même fichier pour supprimer les comptes :

$local = [ADSI]"WinNT://." $fichier = "C:\testPowerShell\listeCompte.txt" if (Test-Path $fichier) { $colLignes = Get-Content $fichier foreach ($ligne in $colLignes) { $tabCompte = $ligne.Split("/") $nom = $tabCompte[0] $compte = [ADSI]"WinNT://./$nom" if ($compte.path) { $local.delete("user", $nom) Write-Host "$nom supprimé" } else { Write-Host "$nom n'existe pas" } } }
⚠️ Avertissement : La suppression est irréversible. En environnement de production, il est conseillé de d'abord désactiver les comptes avant de les supprimer.

4. Interrogation du système Windows

WMI (Windows Management Instrumentation) et CIM (Common Information Model) permettent d'interroger presque tous les aspects du système : matériel, logiciel, configuration, etc.

💡 Get-CimInstance vs Get-WmiObject : Get-CimInstance est plus récent et remplace Get-WmiObject depuis PowerShell 3.0. Il utilise le protocole WS-Management, plus moderne et sécurisé.

4.1. Informations sur le système d'exploitation

Nous avons utilisé CIM pour interroger la classe Win32_OperatingSystem :

$os = Get-CimInstance -ClassName Win32_OperatingSystem Write-Host "Nom du système d'exploitation: $($os.Caption)" Write-Host "Version: $($os.Version)" Write-Host "Architecture: $($os.OSArchitecture)" Write-Host "Date d'installation: $($os.InstallDate)" Write-Host "Mémoire totale (en Mo): $($os.TotalVisibleMemorySize)" Write-Host "Mémoire libre (en Mo): $($os.FreePhysicalMemory)"

Exemple de sortie obtenue :

Nom du système d'exploitation: Microsoft Windows 11 Pro Version: 10.0.22621 Architecture: 64-bit Date d'installation: 20250315120000.000000+060 Mémoire totale (en Mo): 16284 Mémoire libre (en Mo): 8452

4.2. Informations sur le contrôleur vidéo

Classe Win32_VideoController :

$video = Get-CimInstance -ClassName Win32_VideoController Write-Host "Nom du contrôleur vidéo: $($video.Name)" Write-Host "Version du pilote: $($video.DriverVersion)" Write-Host "Mode vidéo (résolution): $($video.VideoModeDescription)" Write-Host "Processeur vidéo: $($video.AdapterCompatibility)" Write-Host "Mémoire vidéo (en octets): $($video.AdapterRAM)"
💡 Remarque : Sur les systèmes avec plusieurs cartes graphiques (ex: ordinateur portable avec Intel + NVIDIA), Get-CimInstance Win32_VideoController retourne plusieurs objets. Il faudrait peut-être filtrer selon l'adaptateur utilisé.

4.3. Informations sur les disques logiques

Classe Win32_LogicalDisk :

$vol = Get-CimInstance -ClassName Win32_LogicalDisk $firstDisk = $vol[0] # Premier disque (généralement C:) Write-Host "Nom du volume: $($firstDisk.VolumeName)" Write-Host "Taille (en Go): $([math]::round($firstDisk.Size / 1GB, 2))" Write-Host "Espace libre (en Go): $([math]::round($firstDisk.FreeSpace / 1GB, 2))" Write-Host "Système de fichiers: $($firstDisk.FileSystem)"
💡 Explication du calcul : la taille est en octets. Division par 1GB (soit 1 073 741 824) pour obtenir des gigaoctets. La fonction [math]::round() arrondit à deux décimales.

6. Conclusion

Ce TP a été une introduction complète à PowerShell pour l'administration système sous Windows.

Ce que nous avons appris :

  1. Fondamentaux du shell : navigation, création/suppression de fichiers et dossiers, manipulation de variables.
  2. Automatisation de la gestion des comptes locaux via ADSI :
    • Création interactive et massive.
    • Suppression massive.
    • Lecture des informations utilisateur.
  3. Interrogation du système avec WMI/CIM pour récupérer :
    • Les caractéristiques du système d'exploitation.
    • Les informations matérielles (carte vidéo, disques).
  4. Bonnes pratiques : tester l'existence (Test-Path), vérifier la présence d'un compte avant création, utiliser -WhatIf avec précaution.

Points forts de PowerShell retenus :

  • La puissance des objets : toute commande retourne des objets, pas du texte brut. On peut alors explorer leurs propriétés avec Get-Member.
  • L'intégration native avec Windows : ADSI et WMI/CIM permettent de gérer presque tous les aspects du système.
  • La répétabilité : un script PowerShell peut être exécuté à l'identique sur des centaines de machines.

Applications concrètes envisagées :

  • Script d'intégration de nouveaux utilisateurs (création du compte, du dossier personnel, des droits).
  • Inventaire matériel automatisé sur un parc de machines.
  • Nettoyage de comptes inactifs (lecture de LastLogin et suppression automatisée).

Limites et précautions :

  • La suppression avec Remove-Item -Recurse -Force est dangereuse : toujours tester avec -WhatIf d'abord.
  • Les mots de passe en clair dans les scripts sont une mauvaise pratique. Pour la production, utiliser des SecureString ou un gestionnaire de secrets.
  • ADSI peut être plus lent que les cmdlets natives sur les versions récentes de Windows.
