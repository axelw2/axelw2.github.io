1. Introduction et objectifs

SNMP (Simple Network Management Protocol) est un protocole standard de l'IETF utilisé pour la supervision et l'administration des équipements réseau (routeurs, switches, serveurs, imprimantes, etc.). Il permet à un gestionnaire central (serveur de supervision) d'interroger des agents SNMP pour récupérer des informations (charge CPU, espace disque, état des interfaces, etc.) et de recevoir des notifications en cas d'incident.

Objectifs du TP :

  1. Installer l'agent SNMP (snmpd) sur une machine Debian.
  2. Configurer le fichier snmpd.conf pour définir les communautés, les vues et les seuils de surveillance.
  3. Comprendre les directives de configuration : rocommunity, sysLocation, disk, load, proc, etc.
  4. Corriger le problème système sur Debian 12 où les modules DisMan/Event MIB sont désactivés par défaut.
  5. Activer l'envoi de traps (notifications) pour les changements d'état des interfaces.
  6. Tester et valider le bon fonctionnement de l'agent SNMP.

2. Qu'est-ce que SNMP ?

SNMP repose sur une architecture client-serveur :

ComposantRôleExemple
Agent SNMPS'exécute sur l'équipement supervisé, collecte les informations et répond aux requêtessnmpd sur Linux
Gestionnaire SNMPInterroge les agents, reçoit les traps, centralise la supervisionZabbix, Nagios, PRTG, SolarWinds
MIB (Management Information Base)Base de données hiérarchique des objets supervisés, identifiés par des OID (Object Identifiers).1.3.6.1.2.1.1.1.0 = description du système

Les opérations SNMP principales :

  • GET : le gestionnaire demande une valeur précise.
  • GETNEXT : parcours d'une table.
  • WALK : récupération récursive d'un sous-arbre.
  • SET : modification d'une valeur (configuration à distance).
  • TRAP : notification asynchrone envoyée par l'agent au gestionnaire.

Versions de SNMP :

VersionSécuritéUtilisation
SNMPv1Communautaire (texte clair)Obsolète, sauf rétrocompatibilité
SNMPv2cCommunautaire (texte clair, mais plus d'opérations)Encore très répandue en interne
SNMPv3Authentification et chiffrement (USM)Recommandé pour la production
💡 Note : Dans ce TP, nous utilisons SNMPv2c avec communauté test123.

3. Installation des paquets SNMP

Sur la machine Debian (serveur supervisé), nous avons installé les paquets suivants :

sudo apt update sudo apt install -y snmp snmp-mibs-downloader snmpd
PaquetRôle
snmpdAgent SNMP (le démon qui tourne sur la machine supervisée)
snmpOutils clients (snmpget, snmpwalk, snmptrap) pour tester
snmp-mibs-downloaderTélécharge les MIBs standard pour faciliter la lecture des OIDs

Vérification de l'installation :

dpkg -l | grep snmp systemctl status snmpd
✅ Par défaut, snmpd est installé mais pas encore configuré.

4. Configuration de base de snmpd

4.1. Sauvegarde et création du fichier de configuration

Le fichier de configuration principal est /etc/snmp/snmpd.conf. Nous avons commencé par sauvegarder la configuration par défaut :

sudo mv /etc/snmp/snmpd.conf /etc/snmp/snmpd.conf.orig

Puis nous avons créé notre propre configuration :

sudo cat > /etc/snmp/snmpd.conf <

Redémarrage du service :

sudo systemctl restart snmpd sudo systemctl status snmpd

4.2. Explication des directives principales

Directive agentAddress

Définit l'adresse et le port sur lequel l'agent SNMP écoute :

agentAddress udp:161,udp6:[::1]:161
  • udp:161 : écoute sur IPv4, port 161 (port standard SNMP).
  • udp6:[::1]:161 : écoute sur IPv6 localhost.

Directive view (vue)

view all included .1

Définit une vue nommée all qui inclut l'intégralité de l'arbre OID à partir de la racine .1 (soit iso.org). Sans cette vue, l'agent ne renverrait aucune information.

Directive rocommunity (communauté lecture seule)

rocommunity test123 default -V all
  • rocommunity : communauté en lecture seule.
  • test123 : nom de la communauté (à remplacer par un mot de passe robuste en production).
  • default : accepte les requêtes de toutes les adresses IP.
  • -V all : applique la vue all (c'est-à-dire autorise l'accès à tout l'arbre OID).
⚠️ Bonne pratique : En production, on limitera l'accès à l'adresse IP du serveur de supervision uniquement : rocommunity test123 192.168.1.50

Directive sysLocation

sysLocation Salle 1 Baie 1 Empl 1

Information textuelle sur l'emplacement physique de la machine (retournée dans les OIDs .1.3.6.1.2.1.1.6.0).

Directive sysContact

sysContact admin <admin@domaine.tld>

Coordonnées de la personne responsable (retournée dans .1.3.6.1.2.1.1.4.0).

Surveillance des processus (proc)

proc mountd proc ntalkd 4 proc sendmail 10 1
  • proc mountd : vérifie qu'au moins un processus mountd tourne.
  • proc ntalkd 4 : attend exactement 4 instances de ntalkd.
  • proc sendmail 10 1 : attend entre 10 et 1 instance (inverse volontaire pour déclencher une alerte).

Surveillance des disques (disk)

disk / 10000 disk /var 5% includeAllDisks 10%
  • disk / 10000 : alerte si l'espace libre sur / passe sous 10 000 Ko (~10 Mo).
  • disk /var 5% : alerte si l'espace libre sur /var passe sous 5%.
  • includeAllDisks 10% : applique un seuil de 10% à tous les autres points de montage.

Surveillance de la charge CPU (load)

load 12 10 5
SeuilPériodeSignification
121 minuteCharge > 12 pendant 1 min
105 minutesCharge > 10 pendant 5 min
515 minutesCharge > 5 pendant 15 min
💡 Explication : Sur une machine monoprocesseur, une charge moyenne de 1 signifie que le CPU est utilisé à 100%. Sur 8 cœurs, la charge max soutenable est d'environ 8.

5. Correction du problème systemd (Debian 12)

5.1. Diagnostic du problème

Sur Debian 12, nous avons rencontré une difficulté : les directives linkUpDownNotifications et defaultMonitors étaient ignorées avec des erreurs du type Unknown token.

Cause identifiée : L'unité systemd de snmpd lance le service avec l'option -I -smux,mteTrigger,mteTriggerConf qui désactive explicitement les modules DisMan/Event MIB (mteTrigger, mteTriggerConf), nécessaires pour les notifications et moniteurs par défaut.

Vérification de la commande actuelle :

systemctl cat snmpd | grep ExecStart
❌ Sortie observée : ExecStart=/usr/sbin/snmpd -LOw -u Debian-snmp -g Debian-snmp -I -smux,mteTrigger,mteTriggerConf -f

Les modules mteTrigger et mteTriggerConf sont exclus (-I = "exclude these modules").

5.2. Modification de l'unité systemd

Pour corriger ce comportement, nous avons modifié l'unité systemd en la personnalisant :

sudo systemctl edit --full snmpd

Dans l'éditeur, nous avons repéré et modifié la ligne ExecStart :

Avant :

ExecStart=/usr/sbin/snmpd -LOw -u Debian-snmp -g Debian-snmp -I -smux,mteTrigger,mteTriggerConf -f

Après (correction) :

ExecStart=/usr/sbin/snmpd -LOw -u Debian-snmp -g Debian-snmp -I -smux -f
💡 Explication : On supprime mteTrigger,mteTriggerConf de l'exclusion. On conserve seulement -smux (SMUX est un ancien protocole rarement utilisé).

Rechargement de systemd et redémarrage :

sudo systemctl daemon-reload sudo systemctl restart snmpd sudo systemctl status snmpd --no-pager

Vérification que les logs ne contiennent plus d'erreurs :

journalctl -u snmpd -b | grep -E "linkUp|linkDown|monitor|Unknown"
✅ Plus aucune erreur Unknown token.

6. Activation des traps (notifications)

Une fois la correction systemd appliquée, nous avons activé les notifications SNMP (traps).

Ajout des directives dans /etc/snmp/snmpd.conf :

sudo nano /etc/snmp/snmpd.conf

Ajout à la fin du fichier :

linkUpDownNotifications yes defaultMonitors yes trapsess -v 2c -c test123 192.168.1.50
DirectiveRôle
linkUpDownNotifications yesEnvoie une notification linkUp ou linkDown à chaque changement d'état d'une interface réseau
defaultMonitors yesActive les moniteurs par défaut (CPU, disques, processus) définis précédemment
trapsess -v 2c -c test123 192.168.1.50Définit la destination des traps : version SNMPv2c, communauté test123, IP du gestionnaire 192.168.1.50

Redémarrage :

sudo systemctl restart snmpd

7. Validation et tests

7.1. Vérification du service

sudo systemctl status snmpd --no-pager sudo ss -tuln | grep 161
✅ Le service doit être active (running) et écouter sur le port UDP 161.

7.2. Consultation des OIDs

Requête de base :

snmpget -v 2c -c test123 localhost sysDescr.0
✅ Sortie attendue : SNMPv2-MIB::sysDescr.0 = STRING: Linux debian 6.1.0-13-amd64 ...

Parcours complet de l'arbre système :

snmpwalk -v 2c -c test123 localhost system

Consultation de la charge CPU (moyenne sur 1, 5, 15 minutes) :

snmpwalk -v 2c -c test123 localhost .1.3.6.1.4.1.2021.10

Consultation de l'espace disque :

snmpwalk -v 2c -c test123 localhost .1.3.6.1.2.1.25.2.3.1

7.3. Simulation d'événements

Simuler un processus manquant :

pkill mountd # si mountd n'est pas censé tourner snmpwalk -v 2c -c test123 localhost .1.3.6.1.4.1.2021.2
💡 Explication : La table prTable indiquera le nombre d'instances trouvées vs attendues.

Simuler un changement d'état d'interface :

sudo ip link set eth0 down sudo ip link set eth0 up
✅ Les traps linkDown et linkUp doivent être envoyés au gestionnaire.

Visualiser les traps localement (en attendant le gestionnaire) :

sudo tcpdump -i eth0 udp port 162 -n

8. Problèmes rencontrés et solutions

ProblèmeCauseSolution
snmpd refuse de démarrer (Unknown token)Les directives linkUpDownNotifications ou defaultMonitors ne sont pas reconnuesModifier l'unité systemd pour supprimer mteTrigger,mteTriggerConf de l'exclusion (-I)
Aucune donnée retournée par snmpwalkLa communauté est incorrecte ou la vue n'est pas définieVérifier rocommunity et view all. Tester avec snmpget -c public localhost sysDescr.0
Les traps n'arrivent pastrapsess mal configuré ou firewall bloquant le port UDP 162Ajouter trapsess -v 2c -c test123 <IP_gestionnaire>. Vérifier qu'UDP 162 est ouvert
Erreur "Timeout" lors des requêtessnmpd n'écoute pas sur l'interface réseauVérifier agentAddress. Par défaut, snmpd écoute sur localhost uniquement. Modifier pour udp:161
Les MIBs ne sont pas résolues (noms au lieu d'OIDs)Paquet snmp-mibs-downloader non installé ou cache MIB non chargéInstaller le paquet, puis modifier /etc/snmp/snmp.conf pour commenter la ligne mibs :
⚠️ Cas particulier : Sur Debian 12, après modification de l'unité systemd, ne pas oublier systemctl daemon-reload, sinon les changements ne sont pas pris en compte.

10. Conclusion

Ce TP nous a permis de déployer un agent SNMP complet sur une machine Debian, prêt à être supervisé par un gestionnaire central (Zabbix, Nagios, etc.).

Ce que nous avons appris :

  1. Architecture SNMP : distinction entre agent (snmpd) et gestionnaire, rôle des MIBs et des OIDs.
  2. Configuration fine de snmpd.conf :
    • Définition des communautés (rocommunity) et des vues (view).
    • Mise en place de seuils de surveillance (processus, disques, charge CPU).
    • Activation des notifications (linkUpDown, defaultMonitors).
  3. Correction d'un problème spécifique à Debian 12 : désactivation par défaut des modules DisMan/Event MIB dans l'unité systemd.
  4. Tests avec les outils clients : snmpget, snmpwalk.
  5. Compréhension des traps : notifications asynchrones pour les événements critiques.

Points de vigilance pour la production :

Bonne pratiquePourquoi
Changer la communauté par défaut (public/private)Éviter la découverte facile par un attaquant
Limiter l'accès à l'IP du serveur de supervisionRéduire la surface d'attaque
Utiliser SNMPv3 si possibleChiffrement et authentification, contrairement à v2c (texte clair)
Ne pas exposer SNMP sur InternetProtocole bavard, vulnérable aux attaques de type "community string guessing"
Restreindre les vues (view)Ne donner accès qu'aux informations nécessaires, pas à tout l'arbre .1
Tester les seuils (disk, load, proc)Éviter les faux positifs ou les seuils trop hauts qui ne déclenchent jamais
